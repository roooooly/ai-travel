## 一、工期估算（关键调整）

### 1.1 原估算 vs 单人现实

| 对比项 | 原估算（3人团队） | 单人全职(8h/天) | 单人兼职(4h/天) |
|--------|------------------|----------------|----------------|
| 总工时 | 211h | 211h | 211h |
| 实际工期 | ~2周 | **~5周** | **~10周** |
| 因素 | 3人并行 | 无并行+上下文切换 | 大量上下文切换 |

### 1.2 单人开发损耗因子

| 损耗来源 | 影响 | 说明 |
|---------|------|------|
| 上下文切换 | +30%时间 | 前端/后端/AI/DB切换损耗 |
| 问题定位 | +20%时间 | 无人讨论，调试慢 |
| 代码质量 | +15%时间 | 无code review，返工多 |
| 数据准备 | 固定成本 | 无法并行，必须人工做 |
| **总计损耗** | **+65%** | 211h × 1.65 ≈ 350h |

### 1.3 现实工期预测

```
单人兼职(4h/天) = 350h ÷ 4h ≈ 87天 ≈ 17周 ≈ 4个月
单人全职(8h/天) = 350h ÷ 8h ≈ 44天 ≈ 9周 ≈ 2个月
```

**结论：必须精简MVP范围！**

## 二、MVP精简方案

### 2.1 精简原则

1. **只做核心链路**：目的地→景点→AI行程→查看/编辑
2. **砍掉复杂功能**：微信登录→简单登录，砍掉打卡成就
3. **砍掉非必要模块**：天气/导航/美食模块延后
4. **砍掉3城市**：MVP只做苏州1个城市
5. **砍掉CI/CD**：本地手动部署

## 八、建议与总结

### 8.1 给Crispin的建议

1. **时间预期**：单人兼职开发MVP = 4-5个月，不要乐观
2. **范围控制**：严格按第一阶段12个任务执行，不要Scope Creep
3. **数据优先**：数据是核心壁垒，建议投入40%以上时间
4. **技术栈**：保持当前推荐（FastAPI + Next.js + PostgreSQL），不要换
5. **上线时机**：本地MVP跑通后再考虑上云，不要急于部署

### 8.2 最低可行性目标

**如果只能投入最少量时间（每天2h）：**

```
总工期 = 165h ÷ 2h ≈ 83天 ≈ 16周 ≈ 4个月

最低目标：
- 核心API（目的地+景点）
- AI行程生成
- 苏州20个景点
= 可演示的Demo
```

### 8.3 推荐投入

**建议每天投入4h以上：**

```

每天6h = 12周完成MVP
```
